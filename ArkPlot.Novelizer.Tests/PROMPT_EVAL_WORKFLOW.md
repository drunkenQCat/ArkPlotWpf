# Novelizer 提示词打擂台评测工作流

## 概述

**模式**：全自动打擂台（Champion Challenge）— 无人值守，连续运行。

每轮由当前 champion 迎战新变体，胜者留任。我自主完成全部步骤：分析短板 → 设计挑战者 → 跑测试 → 锐评 → 判定胜负 → 更新排行榜 → 进入下一轮。无需用户介入。

**预期产出**（N 轮结束后）：
1. 累积排行榜 `RANKING.md` — 每个 prompt 的评分、胜负关系、关键改进点
2. 最终 champion — 替换 `NovelizerPipeline.DefaultSystemPrompt`
3. 演进轨迹 — 从 baseline 到最终版，每步改了什么、为什么改、效果如何

## 全自动循环

用户只需说"跑一轮"或"开始评测"，以下流程自主执行，无需用户确认：

```
┌──────────────────────────────────────────────────┐
│ 1. 分析当前 champion 输出的短板                    │
│    → 逐维度检查（6 个维度），找出最弱的 1-2 个维度  │
│                                                    │
│ 2. 设计挑战者 prompt（参见"改进策略指南"）          │
│    → 只改 1 个维度，只加/改 1 条精准指令            │
│    → 保留 champion 做得好的部分不变                  │
│    → 查阅 RANKING.md 历史，不重复已失败的改进方向   │
│    → 写入 prompts/{NN}_{简短描述}.md                │
│                                                    │
│ 3. 预处理输入（仅首次，后续复用缓存）               │
│    → MarkdownBuilder.PreprocessMdContent(rawMd)    │
│                                                    │
│ 4. 调 LLM                                         │
│    → BailianClient.ChatAsync(flash, prompt, text)  │
│    → 输出写入 output/{name}_flash.md               │
│    → champion 已有输出则跳过（省钱）                 │
│                                                    │
│ 5. 锐评                                           │
│    → 逐篇阅读 champion 和挑战者输出                 │
│    → 按 6 个维度分别打分（1-10）                     │
│    → 计算加权总分                                    │
│    → 写出对比分析                                   │
│                                                    │
│ 6. 判定胜负                                        │
│    → 总分高者胜出                                    │
│    → 总分相同时 champion 留任（守擂优势）            │
│    → 胜者成为新 champion                            │
│                                                    │
│ 7. 更新 RANKING.md                                 │
│    → 追加本轮记录                                   │
│                                                    │
│ 8. 自动进入下一轮 → 回到步骤 1                      │
└──────────────────────────────────────────────────┘
```

## 锐评维度（每项 1-10 分）

| 维度 | 权重 | 评分标准 |
|------|------|---------|
| 信息完整度 | 25% | 原对话是否全部覆盖，有无遗漏/压缩/摘要化；角色标签是否正确（买主卖主是否混淆） |
| 文学质量 | 25% | 段落结构、句式变化（句首是否多样）、节奏感；是否有网文味；比喻是否为场景原创 |
| 角色声音区分度 | 15% | 不同角色的台词/行为是否有辨识度；对话前后是否有独特的物理动作包裹 |
| 场景融入 | 15% | 立绘/场景描述是否自然嵌入叙事；是否清除了"站在虚空中"等元描述；肖像是否碎片化嵌入而非整段粘贴 |
| 去模板化 | 10% | 有无"顿了顿"、"轻轻叹气"、"目光交汇"等万能动作滥用；"仿佛"使用次数 |
| 声音/音乐处理 | 10% | BGM/音效是否转化为叙事张力，而非直接提及"背景音乐响起" |

**总分** = 加权平均。

**守擂优势**：总分相同时 champion 留任，避免无意义的冠军交替。

**输出长度参考**：同等质量下，更长（更多动作包裹和场景细节）的输出通常信息完整度更高。明显缩短（< champion 的 80%）应扣分。

## 改进策略指南

> 以下规则来自首轮 10 轮打擂台的实战经验，后续评测必须遵守。

### 铁律：每轮只改一处

每轮挑战者只改 **1 条** 精准指令。不要在一轮中同时修改多个维度。

- 改太多 → 模型注意力分散 → 某条指令被忽略 → 退步
- 改一条 → 效果可归因 → 能看清哪个改动有效/无效

### 禁止：黑名单策略

不要在 prompt 中放大量"不要做什么"的禁用表达清单。

- 模型对"不要 X"的遵从度不稳定，经常无视
- 过多的禁令让模型变得保守，失去原创灵气
- 实战：黑名单从 3 条扩到 9 条后，总分反而下降

**替代方案**：如果要减少某个表达，只放 1 条限制（如"全文最多使用 N 次"），而非列出全部禁词。

### 禁止：正面示例

不要在 prompt 中放"优秀示范"段落。

- 模型会把示例原封不动地搬进输出（复读机效应）
- 实战：放了"声音沙哑如从炉灰里捞出来"作为示例，输出中一字不差地出现了

**替代方案**：用抽象的规则描述（如"比喻必须是场景特定的"），而非具体句子。

### 聚焦 vs 泛化

改写指令应聚焦于特定类型的文本，而非要求改写一切。

- "改写肖像描写" → ✅ 有效，描写质量显著提升
- "改写所有描述性文本" → ❌ 无效，输出变短，灵气丧失

## ⚠️ 关键发现：模型方差是最大敌人

> 25 轮实验的最大发现：**deepseek-v4-flash 在此输入上存在巨大输出方差，prompt 工程无法解决。**

### 方差数据

| Prompt | 总运行次数 | 全覆盖次数 | 覆盖率 | 典型输出 |
|--------|:---:|:---:|:---:|:---:|
| 09 (champion) | 7 | 1 | 14.3% | ~2200 字符 |
| 00 (baseline) | 3 | 0 | 0% | ~2600 字符 |

**两个 prompt 都无法稳定覆盖全部场景。** 原始 09 的 5574 字符全覆盖输出是一次统计异常值（7 次中仅出现 1 次）。

### 对评测流程的影响

1. **单次评测不可靠** — 必须至少跑 3 次取中位数/最长输出
2. **评分必须标注方差** — 任何基于单次运行的评分都是噪声
3. **比较 prompt 时需控制变量** — 每个 prompt 至少跑 3 次，用覆盖率作为首要筛选指标
4. **覆盖率 > 文学质量** — 跳过场景的输出无论多好看都是不合格的

### 推荐的架构层面修复方案

1. **分场景调用**（推荐）— 先用 ChapterSplitter 拆分场景，每个场景独立调 LLM，最后拼接
2. **多次运行取最长** — 同一 prompt 跑 3-5 次，取字符数最长的输出
3. **预处理增强** — 在 PreprocessMdContent 中保留更多场景标识
4. **升级模型** — 换用 deepseek-v4-pro 或其他方差更小的模型

## 自主改进策略

每轮结束后，我自主执行以下分析，不等待用户指导：

1. **短板定位**：找到 champion 得分最低且与满分差距最大的维度
2. **失败归因**：对比输入原文和 champion 输出，定位具体哪些段落/句子出了问题
3. **策略校验**：对照上方"改进策略指南"，确认计划的改动不踩已知的坑
4. **精准改写**：只写 **1 条** 精准指令来解决该短板，不动其他部分
5. **避免回退**：查阅 RANKING.md 历史记录，不重复已尝试过但失败的方向

## 最终收尾

N 轮跑完（或用户喊停）后：

1. 将最终 champion 内容写入 `NovelizerPipeline.DefaultSystemPrompt`
2. `RANKING.md` 保持完整存档
3. 所有 `prompts/` 和 `output/` 文件保留，供日后回溯
